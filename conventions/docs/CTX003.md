# CTX003 — `CtxId` validates convention identifiers at compile time

The xtask convention checker uses [`CtxId`](../../xtask/src/check_id.rs) as
the canonical type for a CTX identifier (the stable code surfaced in
diagnostic output, the filename in `conventions/docs/`, and the value
returned by `Check::id()`). `CtxId` has no public string constructor —
the only way to obtain one is `CtxId::new("CTX###")`, which validates the
format inside a `const` block.

## Why

`CtxId` is the canonical example in this codebase of a Tier 1 rule: the
wrong code does not compile because the type to express it does not exist.
There is no diagnostic to author, no doc to maintain, no fitness test to
keep in sync. The type definition *is* the enforcement.

This matters specifically for `Check::id()` because:

- Every CTX id appears in three places (the `Check` impl, the diagnostic
  emitted at runtime, and the matching `conventions/docs/<id>.md`
  filename). A typo at the trait level would propagate silently.
- The drift self-test validates that `Check::id()` matches frontmatter
  `id`, but it can only catch *registered* mismatches. A misspelled check
  whose doc was also misspelled would slip through. `CtxId::new` rejects
  malformed strings at the call site instead.

See [ADR-0004](../adr/ADR-0004-ctxid-newtype.md) for rationale.

## How violations look

Using a literal that doesn't match `^CTX[0-9]+$`:

```rust,compile_fail
use xtask::CtxId;
const ID: CtxId = CtxId::new("ctx001");      // lowercase
const ID: CtxId = CtxId::new("RULE001");     // wrong prefix
const ID: CtxId = CtxId::new("CTX-DEPS");    // hyphen rejected
const ID: CtxId = CtxId::new("CTX");         // no digits
```

Each fails with `evaluation of `<expr>` failed` at the const evaluation
site, surfaced as a normal compile error.

## How to use

```rust
use xtask::CtxId;

const ID: CtxId = CtxId::new("CTX042");

impl Check for MyCheck {
    fn id(&self) -> CtxId {
        ID
    }
    // ...
}
```

## Enforcement

- **Tier:** 1 (type system).
- **Implementation:** `xtask/src/check_id.rs`.
- **Diagnostic code:** none — the rule is enforced by `rustc`.
- **Doctest:** `compile_fail` doctests in `CtxId::new` lock in the
  validator. If a regression weakens the bound (for example by changing
  the `assert!` to an `unreachable_unchecked`), the doctest will start
  *succeeding* and the `compile_fail` annotation will fail the build.
- **Promotion path:** stays at Tier 1.

## Note on `CTX-DEPS` and other doc-only ids

Convention docs that aren't backed by a registered xtask check (e.g.
`CTX-DEPS` for cargo-deny, future Tier 1 docs that just describe a type)
are not constrained to the `CtxId` regex. Only ids that flow through
`Check::id()` need to be valid `CtxId`s.
